# DLBCL 二分型训练效果问题分析与修改建议

## 1. 文档目的

本文档用于回答三个问题：

1. 当前项目里，为什么 `nanchang`、`morph`、`all` 等数据集的训练效果不稳定，或者总体效果不理想。
2. 这些问题分别属于数据问题、评估设计问题、训练流程问题，还是模型本身问题。
3. 如果要修改项目，应该优先改哪些文件、哪些函数、哪些参数，以及改完之后怎么验证修改是否真的有效。

本文档只给出理论分析和修改方案，不直接修改训练代码。

## 3. 先说结论

当前模型效果不理想，不是单一 bug 造成的，而是以下问题叠加：

1. `nanchang` 的 10 折评估设计过于激进，每折验证集和测试集太小，导致单折 AUC 大幅波动。
2. 训练阶段的 early stopping 目前按 `val_loss` 选 checkpoint，而不是按 `val_auc`，与研究目标不一致。
3. 多个实验中训练误差都能降到 0，说明模型已经把训练集记住了，但泛化没有同步提高，存在明显过拟合。
4. `all` 数据集混合了 `nanchang`、`morph`、`tcga` 三个来源，但当前 split 没有做 source-aware 分层，跨域偏移被直接放大。
5. 当前 DLBCL 数据集整体并不严重失衡，`weighted_sample` 很可能是在放大小样本噪声，而不是解决类别失衡。
6. 交叉验证的作用是估计模型对不同患者划分的泛化能力，不是让我们从 10 折里挑一折最好的当最终结论。

一句话概括：

当前项目的核心问题，不是“模型完全学不会 DLBCL 二分型”，而是“模型能学到信号，但实验设计和选模逻辑把方差、过拟合和跨域偏移放大了”。

## 4. 10 折交叉验证为什么和单次训练不一样

这是整个项目里最容易被误解的地方，必须先讲清楚。

### 4.1 单次训练是什么

单次训练通常是：

1. 先固定一次 `train/val/test` 划分。
2. 训练一个模型。
3. 在这个固定测试集上评估一次结果。

这种方式得到的是“这个特定划分下”的表现。

### 4.2 10 折交叉验证是什么

10 折交叉验证不是简单地“换一个测试集对象”。

它实际做的是：

1. 生成 10 组不同的病人级别划分。
2. 每一折都重新初始化模型。
3. 每一折都重新训练。
4. 每一折都独立 early stop。
5. 每一折都独立保存自己的 checkpoint。
6. 最后把 10 个测试结果做汇总。

所以 10 折结果回答的问题是：

“如果病人划分发生变化，这个方法还能不能稳定泛化？”

这和单次训练回答的问题不同。

### 4.3 为什么不能直接选最好的那一折

如果目标是做“方法准确性”的研究，不能把 10 折里最好的一折当成最终性能，因为：

1. 最好的一折可能只是划分刚好更容易。
2. 如果你看完测试结果再选最好的那一折，本质上是在用测试集做模型筛选。
3. 这样会高估模型的真实性能。

更合理的做法是：

- 准确性：报告 `10-fold mean ± std`，或报告 OOF 汇总结果。
- 可解释性：可以选“验证集表现较好”的代表 fold 做热图展示，但不能只展示测试集最好的成功案例。

## 5. 当前数据集和结果概况

### 5.1 数据规模

当前仓库里的 DLBCL 数据集大致如下：

| 数据集 | slides | cases | 说明 |
|---|---:|---:|---|
| `nanchang` | 389 | 50 | 单院数据，病例数很少 |
| `morph` | 183 | 132 | 由 IHC + Hans 算法推导标签 |
| `tcga` | 39 | 39 | 病例数更少 |
| `all` | 611 | 221 | `nanchang + morph + tcga` 混合 |

### 5.2 现有实验结果

从 `results/*/summary.csv` 看，当前关键实验结果大致如下：

| 实验目录 | 数据集 | 模型 | mean test AUC | mean test Acc |
|---|---|---|---:|---:|
| `results/nanchang_uni_clam_sb_s1` | nanchang | UNI + CLAM_SB | 0.6599 | 0.5754 |
| `results/nanchang_uni_mil_s1` | nanchang | UNI + MIL | 0.5767 | 0.5455 |
| `results/morph_uni_clam_sb_s1` | morph | UNI + CLAM_SB | 0.6568 | 0.5596 |
| `results/dlbcl_all_uni_clam_sb_s1` | all | UNI + CLAM_SB | 0.5367 | 0.4876 |
| `results/dlbcl_all_clam_sb_s1` | all | ResNet + CLAM_SB | 0.4327 | 0.4845 |
| `results/dlbcl_all_clam_mb_s1` | all | ResNet + CLAM_MB | 0.3740 | 0.4587 |

这些结果说明：

1. 单独数据集 `nanchang` 和 `morph` 不是完全没信号，AUC 都大约在 `0.65` 左右。
2. `MIL` 低于 `CLAM_SB`，说明 CLAM 的 attention 机制有一定帮助。
3. `all` 数据集明显更难，混合来源以后性能大幅下降。
4. 当前结论不应该是“CLAM 完全不行”，而应该是“CLAM 能学到一部分信息，但现有训练流程没有把能力稳定释放出来”。

## 6. TensorBoard 告诉了我们什么

基于历史阶段的 `analysis_results/tfevents_fold_summary.csv` 统计，当前训练过程暴露出几个非常关键的问题。

### 6.1 几乎所有关键实验都把训练误差降到了 0

以下实验中，`10/10` 个 fold 都出现了 `min_train_error == 0`：

- `nanchang_uni_clam_sb_s1`
- `morph_uni_clam_sb_s1`
- `dlbcl_all_uni_clam_sb_s1`
- `nanchang_uni_mil_s1`

这说明：

1. 特征里确实有信号。
2. 模型容量足够。
3. 训练集被高度记忆。
4. 当前主要矛盾已经不是“欠拟合”，而是“过拟合后泛化不足”。

### 6.2 `best_val_loss_epoch` 和 `best_val_auc_epoch` 经常不一致

根据事件文件统计：

| 实验 | `best_val_loss_epoch <= 1` | `best_val_loss_epoch != best_val_auc_epoch` |
|---|---:|---:|
| `nanchang_uni_clam_sb_s1` | 9/10 | 6/10 |
| `morph_uni_clam_sb_s1` | 1/10 | 10/10 |
| `dlbcl_all_uni_clam_sb_s1` | 9/10 | 7/10 |

这意味着：

1. 很多 fold 在极早期就得到了最小 `val_loss`。
2. 但最好的 `val_auc` 常常出现在别的 epoch。
3. 当前代码按 `val_loss` 选 checkpoint，会导致最终保存的模型不是“分类能力最好的模型”。

这不是小问题，而是当前训练效果不稳定的核心原因之一。

### 6.3 当前 early stopping 更像“默认 50 多个 epoch 后停”，不是“真正根据最优指标选模”

当前 `utils/core_utils.py` 里的 `EarlyStopping` 配置是：

- `patience = 20`
- `stop_epoch = 50`

结合事件文件，很多实验都记录了大约 `52` 个 epoch。

这说明：

1. 当前 early stopping 不是很积极地防止过拟合。
2. 同时，它保存 checkpoint 的依据又是 `val_loss`。
3. 所以当前流程容易出现“训练已经记住训练集，但保存的还不是最优 AUC 模型”的组合问题。

## 7. 各数据集的核心问题

## 7.1 `nanchang` 的问题

### 7.1.1 当前 split 太小

从 `results/nanchang_uni_clam_sb_s1/splits_*.csv` 统计得到：

- 每折平均 `train cases = 42`
- 每折平均 `val cases = 4`
- 每折平均 `test cases = 4`

而且每折验证集和测试集通常是：

- `2 个 GCB`
- `2 个 non-GCB`

这会导致：

1. 单折 AUC 极其敏感。
2. 只要 1 个病例方向变了，整折指标都会显著变化。
3. `0.35 ~ 0.98` 这种单折波动，在这个设置下并不奇怪。

所以 `nanchang` 当前最大的矛盾不是模型本身，而是评估协议太脆弱。

### 7.1.2 过拟合会被这个评估设计进一步放大

`nanchang` 只有 50 个 case，但模型用的是 bag-level + attention + 可选 instance clustering。

理论上会出现下面这种情况：

1. 模型很快记住训练集中与标签相关的 patch 模式。
2. 由于 val set 只有 4 个 case，验证集上的 loss 和 AUC 都非常不稳定。
3. 结果就是：训练误差很低，测试结果高度波动。

因此，`nanchang` 不适合继续用当前这种“10 折 + 极小 val/test”方式作为主结论。

## 7.2 `morph` 的问题

`morph` 的 mean test AUC 约为 `0.6568`，比 `nanchang` 稳定一些，原因不是模型更强，而是：

1. `morph` 的病例数更多。
2. 每折 val/test 的 case 数明显更大。
3. 因而单折评估波动比 `nanchang` 小。

但 `morph` 本身还有一个额外问题：

- 标签来自 Hans 算法推导，不是直接的分子分型金标准。

这意味着它更像“临床代理标签任务”，而不是严格的分子分型标签任务。

因此，`morph` 可以用来做有价值的单独实验，但解释结论时必须明确：

- 它反映的是“形态与 Hans 推导 COO 的关联性”
- 不应直接等价于“真实 COO 分子亚型预测能力”

## 7.3 `all` 的问题

`all` 的当前主要问题不是数据少，而是来源混杂。

`all` 包含：

- `nanchang`: 389 slides / 50 cases
- `morph`: 183 slides / 132 cases
- `tcga`: 39 slides / 39 cases

当前 `all` 的 split 仍然主要是 patient-level，但没有显式约束 source 分布。

这会造成：

1. 某些 fold 的 test set 里几乎没有某个来源。
2. 某些 fold 的测试集来源构成和训练集差异很大。
3. 结果中混入了大量来源偏移效应。

因此 `all` 的 `0.5367` 不能简单理解为“加更多数据反而变差”，更准确的理解是：

“单域任务变成了跨域混合任务，但 split 设计和训练策略没有同步升级。”

## 8. 当前项目的根本问题清单

下面按问题类别给出完整结论。

### 8.1 评估设计问题

1. `nanchang` 的每折 val/test case 太少。
2. 当前研究过度依赖 fold 均值，但没有把 OOF 汇总放到同等重要位置。
3. `all` 没有做 source-aware split。
4. 当前不适合根据“最好的一折”来定义方法性能。

### 8.2 训练流程问题

1. early stopping 按 `val_loss` 选模，而不是按 `val_auc`。
2. 当前只保存一套 checkpoint，信息不够。
3. `weighted_sample` 默认打开时，对当前 DLBCL 数据并不一定有利。
4. `CLAM` 和 `bag-only CLAM` 的消融不够系统。

### 8.3 数据与标签问题

1. `nanchang` 的病例数太少。
2. `morph` 的标签属于代理标签。
3. `all` 混合多来源带来域偏移。
4. 目前缺少严格的 source-level 分析输出。

### 8.4 实验管理问题

1. 你之前对不同数据集使用了不同 splits，但覆盖了旧 split 目录。
2. 虽然训练结果目录里保留了当时的 `splits_*.csv` 副本，历史实验仍然可追溯，但根目录 `splits/` 会误导后续复查。
3. 当前实验输出里没有把“选模依据”和“实际使用的 split 元信息”记录完整。

## 9. 应该怎么改：总原则

如果目标是研究“DLBCL 二分型的准确性和可解释性”，那么正确的修改方向不是盲目换模型，而是：

1. 让评估协议更稳。
2. 让 checkpoint 选择逻辑和研究目标一致。
3. 把单域任务和跨域任务分开解释。
4. 建立清晰的 baseline 和消融。

换句话说，先把实验设计改对，再谈更复杂的方法。

## 11. 按文件给出详细修改建议

## 11.1 [utils/core_utils.py](../utils/core_utils.py)

这是当前最关键的修改文件。

### 11.1.1 为什么必须改这个文件

因为当前训练效果不稳定的核心之一，就是这里的 early stopping 逻辑和你真正关心的 AUC 不一致。

当前现状：

- `EarlyStopping.__call__()` 用的是 `val_loss`
- `validate()` 和 `validate_clam()` 也把 `val_loss` 传给 early stopping
- 最终只保存 `s_{fold}_checkpoint.pt`

这导致的问题：

1. 只能按 loss 选模型。
2. 不能比较 loss-based 和 auc-based 选模差异。
3. 很多 fold 虽然有更好的 `val_auc`，但没有被保留。

### 11.1.2 具体要改什么

建议把 `EarlyStopping` 改成通用监控器，支持以下模式：

- `mode='min'` 用于 `val_loss`
- `mode='max'` 用于 `val_auc`

建议新增字段：

- `monitor_name`
- `mode`
- `best_value`
- `best_epoch`
- `counter`
- `early_stop`

建议行为：

1. 如果监控指标是 `val_loss`，则越小越好。
2. 如果监控指标是 `val_auc`，则越大越好。
3. 保存 checkpoint 时同时写出：
   - 当前最优指标值
   - 当前 epoch
   - 监控指标名称

### 11.1.3 建议新增的 checkpoint 设计

当前只有：

- `s_{fold}_checkpoint.pt`

建议改成：

- `s_{fold}_checkpoint_auc.pt`
- `s_{fold}_checkpoint_loss.pt`

如果只想保留一套默认模型，也至少要：

- 默认保存 `s_{fold}_checkpoint.pt`
- 另外补充 `s_{fold}_checkpoint_loss.pt`

但更推荐两套都保留。

### 11.1.4 建议在 `train()` 中增加的输出

在每个 fold 结束时，建议额外保存一个 `fold_{i}_selection.json` 或 `fold_{i}_selection.csv`，内容至少包括：

- `best_val_auc`
- `best_val_auc_epoch`
- `best_val_loss`
- `best_val_loss_epoch`
- `selected_checkpoint_type`
- `final_test_auc`
- `final_test_acc`

这样后续分析就不再依赖 TensorBoard 事件文件重新回推。

### 11.1.5 修改完成后的验证方法

改完以后，必须验证：

1. 同一 fold 是否能同时生成 `checkpoint_auc` 和 `checkpoint_loss`。
2. `checkpoint_auc` 对应的 epoch 是否真的是 `val/auc` 最大的 epoch。
3. 重跑 1 个小实验后，`summary.csv` 是否能多出最佳 epoch 信息。

## 11.2 [main.py](../main.py)

`main.py` 负责参数入口和训练逻辑组织，必须把训练目标显式化。

### 11.2.1 为什么要改

因为当前 CLI 层面对选模目标不透明，用户运行训练时无法明确指定：

- 到底按 `val_loss` 还是 `val_auc` 选模型
- DLBCL 默认是不是应该关闭 `weighted_sample`
- 当前是否处于标准 CLAM 还是 bag-only CLAM

### 11.2.2 建议新增参数

建议新增：

```bash
--monitor_metric val_auc
```

或

```bash
--early_stop_metric val_auc
```

可选值建议：

- `val_auc`
- `val_loss`

还可以增加：

```bash
--save_best_auc_ckpt
--save_best_loss_ckpt
```

如果不想加太多参数，也可以内置默认规则：

- `task_3_dlbcl_coo` 默认按 `val_auc`
- 其他任务保留 `val_loss`

但从长期维护看，显式参数更清晰。

### 11.2.3 建议调整 DLBCL 任务默认参数

对 `task_3_dlbcl_coo`，建议把推荐默认值改保守：

- `drop_out = 0.5`
- `reg = 1e-3`
- `model_size = small`
- `weighted_sample = False`

原因：

1. 当前训练已经明显过拟合。
2. 你更需要降低方差，而不是提高容量。
3. 这些参数比继续堆复杂结构更符合当前数据规模。

### 11.2.4 建议补充实验输出元信息

当前 `experiment_*.txt` 已经记录了一些参数，但建议补充：

- `monitor_metric`
- `selected_checkpoint_type`
- `split_dir`
- `split_hash` 或 split 描述文件路径
- `dataset`
- `feature_type`

这样你后续回顾实验时，不用再从多个目录倒推。

### 11.2.5 修改完成后的验证方法

1. 运行一次训练命令，确认控制台和实验记录中都能看到 `monitor_metric`。
2. 确认 `results/<exp>/` 下产生了新的 checkpoint 文件。
3. 确认不同参数设置下不会误用旧 checkpoint。

## 11.3 [create_splits_seq.py](../create_splits_seq.py)

这是第二个最重要的文件，因为你当前实验的不稳定性很大程度来自这里。

### 11.3.1 为什么要改

当前脚本已经支持：

- `--dataset`
- `--val_frac`
- `--test_frac`

而且也会生成 dataset-specific 目录，例如：

- `splits/task_3_dlbcl_coo_nanchang_100`

这比老版本已经好很多。

但还不够，因为目前还有两个关键问题：

1. `nanchang` 默认的 `val_frac=0.1`、`test_frac=0.1` 仍然过小。
2. `all` 没有 source-aware 分层。

### 11.3.2 对 `nanchang` 的修改建议

建议二选一：

#### 方案 A：改成 5-fold

适合想让每折 test 更稳的场景。

建议：

- `k = 5`
- `val_frac` 适当上调
- 保证每折 val/test 至少有更充足的 case

#### 方案 B：保留 10-fold，但改变结果解释方式

适合你想保留更高利用率的场景。

建议：

- 仍可保留 10-fold
- 但研究主结论用 OOF 汇总，不主打单折平均
- 同时不再强调单折高低波动

如果必须从代码层面优化，我更推荐方案 A。

### 11.3.3 对 `all` 的修改建议

这是 `all` 当前最需要补的能力。

建议做 source-aware split，即在病人级别划分时，同时考虑：

- `label`
- `source`

理想目标是让每个 fold 的：

- train source 分布
- val source 分布
- test source 分布

都尽量接近整体分布。

### 11.3.4 实现 source-aware split 的建议方式

建议先做最稳妥的版本，不要一开始就写得太复杂：

1. 在 `dataset_csv/dlbcl_all.csv` 增加 `source` 列。
2. 在 split 脚本里读取 `case_id -> source`。
3. 生成联合分层标签，例如：
   - `nanchang_GCB`
   - `nanchang_nonGCB`
   - `morph_GCB`
   - `morph_nonGCB`
   - `tcga_GCB`
   - `tcga_nonGCB`
4. 基于这个联合标签做 case-level split。

如果当前不想动 CSV，也可以在脚本里根据 `slide_id` 或 `case_id` 前缀推断 source，但不推荐长期这么做，因为可维护性差。

### 11.3.5 建议新增 split 产物

每次生成 split 时，除了已有文件外，建议额外输出：

- `split_manifest.json`
- `fold_source_distribution.csv`
- `fold_label_distribution.csv`

其中至少包含：

- 每折 train/val/test 的 case 数
- 每折 label 分布
- 每折 source 分布
- seed
- k
- val_frac
- test_frac

### 11.3.6 修改完成后的验证方法

1. 对 `nanchang` 重新生成 splits，确认每折 val/test case 数符合预期。
2. 对 `all` 重新生成 splits，检查每折 source 比例是否更均衡。
3. 对比旧 split 和新 split 的 fold 分布表，确认改动确实生效。

## 11.4 [eval.py](../eval.py)

虽然当前最核心的问题发生在训练阶段，但评估入口也需要同步改。

### 11.4.1 为什么要改

如果训练阶段开始同时保存：

- `checkpoint_auc`
- `checkpoint_loss`

那么评估脚本就必须允许你明确指定加载哪一种模型。

否则训练端即使改好了，评估端仍然会默认读老逻辑的文件。

### 11.4.2 建议增加的参数

建议新增：

```bash
--ckpt_type auc
```

可选值：

- `auc`
- `loss`
- `default`

对应加载：

- `s_{fold}_checkpoint_auc.pt`
- `s_{fold}_checkpoint_loss.pt`
- `s_{fold}_checkpoint.pt`

### 11.4.3 和 `--auto_fix_inversion` 的关系

当前仓库已经支持 `--auto_fix_inversion`，这是有价值的，但要明确定位：

1. 它是评估阶段的保险措施。
2. 它不能替代训练阶段正确的选模逻辑。
3. 当前主要矛盾不是“所有模型都发生聚类反转”，而是“过拟合 + 选模不一致 + split 太脆弱”。

所以 `auto_fix_inversion` 应保留，但不能当作主修复方案。

## 11.5 [dataset_csv/dlbcl_all.csv](../dataset_csv/dlbcl_all.csv)

### 11.5.1 为什么建议改这个文件

因为 `all` 的最大方法学问题是 source 混合，而 source 目前并不是一个显式字段。

如果没有 `source` 列，后续很多事情都只能靠推断：

- split 分层
- source-level 分析
- per-source 指标汇总

这会让整个实验系统变得脆弱。

### 11.5.2 建议怎么改

在 CSV 中增加一列：

```csv
source
```

取值例如：

- `nanchang`
- `morph`
- `tcga`

这样后续可以直接做：

- source-aware split
- source-wise evaluation
- source-specific visualization

### 11.5.3 这一步的价值

这一步不是为了“提高 AUC 的魔法技巧”，而是为了让 `all` 的研究结论站得住。

## 12. 推荐的 baseline 和消融设计

如果你真想把这个项目做成一篇能自圆其说的研究，模型比较顺序必须改干净。

建议按下面三组做。

### 12.1 Baseline 1：MIL

作用：

- 作为最基础的 bag-level 基线。
- 回答“如果不用 CLAM，只做最简单 MIL，性能会怎样？”

### 12.2 Baseline 2：bag-only CLAM

命令思想：

- `clam_sb`
- `--bag_weight 1.0`
- `--no_inst_cluster`

作用：

- 排除 instance clustering 的影响。
- 回答“CLAM 的 attention + bag-level 分类本身是否已经足够？”

这组非常关键，因为它能直接告诉你：

- 当前问题到底出在 CLAM 主体，还是出在 instance clustering。

### 12.3 Baseline 3：标准 CLAM

作用：

- 与 bag-only 对照，判断 instance clustering 是帮助还是干扰。

### 12.4 不建议现在优先做的方向

当前不建议优先：

1. 继续堆 `clam_mb`
2. 继续增加更复杂结构
3. 先在 `all` 上盲目调参

因为现在连最基础的选模逻辑和 split 设计都没有稳定下来。

## 13. 推荐的训练参数修改方向

下面是理论上更适合当前 DLBCL 数据规模的参数方向。

### 13.1 推荐默认值

对 DLBCL 任务，建议优先试：

- `drop_out = 0.5`
- `reg = 1e-3`
- `model_size = small`
- `weighted_sample = False`

### 13.2 为什么这样设置

因为当前问题主要是：

1. 训练误差过快归零。
2. 验证集指标不稳。
3. `weighted_sample` 对当前近似平衡任务没有明确收益。

这些设置的共同目标是降低方差，而不是增加拟合能力。

### 13.3 `weighted_sample` 为什么不应作为默认

当前 case-level 分布大致是：

| 数据集 | case-level 分布 |
|---|---|
| `nanchang` | GCB = 25, non-GCB = 25 |
| `morph` | GCB = 57, non-GCB = 75 |
| `all` | GCB = 105, non-GCB = 116 |

这不是严重类别失衡。

所以在这种场景下默认开启 `weighted_sample`，更可能带来：

1. 采样噪声放大
2. 每个 epoch 的 bag 组成更不稳定
3. 小数据集上方差进一步增加

因此建议：

- 默认关闭
- 只在确实严重失衡的数据上再单独启用并比较

## 14. 推荐的重跑顺序

不要一次性重跑所有实验，按下面顺序来。

### 第一步：修复训练逻辑后，只重跑最关键的单域实验

推荐先跑：

1. `nanchang + UNI + MIL`
2. `nanchang + UNI + CLAM_SB bag-only`
3. `nanchang + UNI + 标准 CLAM_SB`
4. `morph + UNI + CLAM_SB bag-only`
5. `morph + UNI + 标准 CLAM_SB`

目的：

- 先看 source 单独情况下，选模修正后是否更稳。
- 先回答 instance clustering 是否真的有帮助。

### 第二步：确认单域实验稳定后，再做 `all`

对 `all`：

1. 先重做 source-aware split。
2. 再跑 `all + UNI + bag-only`
3. 再跑 `all + UNI + 标准 CLAM_SB`

如果这一步仍然差，再考虑：

- 域适配
- 来源特定归一化
- leave-one-source-out 设计

### 第三步：补充 seed 稳定性

每组关键实验建议至少跑：

- `3` 个不同 seed

否则你很难证明效果不是偶然。

## 17. 总结

当前项目的问题，不在于“模型完全没法做 DLBCL 二分型”，而在于：

1. `nanchang` 的评估协议太脆弱。
2. checkpoint 选择指标和研究目标不一致。
3. 训练已经明显过拟合。
4. `all` 的跨域混合问题没有被 split 设计正确处理。
5. 实验管理虽然比之前好，但仍缺少足够清晰的元信息记录。

因此，最正确的修复路线不是先换模型，而是：

1. 先把 `AUC-based` 选模改好。
2. 再把 `nanchang` 和 `all` 的 split 设计改好。
3. 再用 `MIL`、`bag-only CLAM`、标准 `CLAM` 做干净消融。
4. 最后再讨论更复杂的方法。

如果后续开始实际改代码，这份文档应作为改动清单执行。

## 18. 聚类反转问题（Clustering Inversion）

### 18.1 问题现象

CLAM 模型在训练时 instance-level clustering 准确率接近 100%，但验证时 AUC 接近 0：

```
训练日志:
  class 0 clustering acc 1.0: correct 2616/2616
  class 1 clustering acc 1.0: correct 2616/2616  ✓ 聚类完美

验证日志:
  Val Set, val_loss: 6.7, val_error: 0.97, auc: 0.0000  ✗ AUC接近0
```

### 18.2 根本原因

Instance-level 聚类正确地将 patch 分成了两类（cluster 0 和 cluster 1），但 bag-level 分类器的权重方向反了：

- **正确情况**: cluster 0 → class 0 (non-GCB), cluster 1 → class 1 (GCB)
- **反转情况**: cluster 0 → class 1 (GCB), cluster 1 → class 0 (non-GCB)

这导致模型预测的概率与实际标签完全相反，AUC = 1 - 真实AUC。

### 18.3 发生概率

根据实验数据分析：
- nanchang 数据集: 约 40% 的 fold 出现聚类反转
- dlbcl_all 数据集: 约 10% 的 fold 出现聚类反转

### 18.4 解决方案

| 方案 | 命令 | 说明 |
|------|------|------|
| MIL 模型 | `--model_type mil` | 无聚类，纯 bag-level 分类 |
| 关闭聚类 | `--bag_weight 1.0 --no_inst_cluster` | 只用 bag loss |
| **自动修正（推荐）** | `--auto_fix_inversion` (eval.py) | 检测到反转时自动翻转预测 |
| 手动翻转 | AUC≈0 时输出 1-AUC | 临时方案 |

### 18.5 使用方法

```bash
# 启用自动翻转评估
python eval.py --auto_fix_inversion --ckpt_type auc --save_exp_code eval_fixed ...
```

**评估逻辑**：当检测到 AUC < 0.3 且翻转后 AUC > 0.7 时自动翻转。

### 18.6 风险评估

| 风险 | 等级 | 说明 | 缓解措施 |
|------|------|------|----------|
| 误判正常为反转 | 低 | 条件为 AUC < 0.3 且翻转 > 0.7，双重条件 | 阈值已设置保守 |
| 漏判反转 | 低 | 只在极端情况下发生 | 可调整阈值 |
| 影响正常模型 | 无 | 只在检测到反转时触发 | 自动条件判断 |
